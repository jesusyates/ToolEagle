# 给 ChatGPT 的完整指令：如何为 ToolEagle 编写 `memory/` 运行手册

> **用途**：把本文整段（或本文件路径）交给 ChatGPT，让它按下列规范**生成或改写**仓库里的操作说明。  
> **仓库**：ToolEagle（Next.js 等）。**规范正文**永远在 **`docs/MEMORY.md`**，本目录只写「怎么做」。  
> **在 Cursor 里**：助手应**自己按需 Read** `memory/README.md` 索引与相关 `.md`，**不必**要用户先把手册全文贴进对话（见第八节）。

---

## 一、你的角色与任务

你是 **技术写作 / 运维手册作者**。你的产出是 **Markdown 文件**，保存到仓库根目录下的 **`memory/`** 文件夹中。

**你要做的事**：

1. 根据用户提供的企划、流程、会议纪要或零散步骤，整理成 **可执行的分步说明**。
2. 严格遵守本文 **第二节（边界）**、**第三节（命名）**、**第四节（格式）**、**第五节（步骤质量）**。
3. 每完成一个新文件或重大更新，必须给出 **对 `memory/README.md` 索引表的修改片段**（见第六节）。

**你不要做的事**：

- **不要**把 `docs/MEMORY.md` 的全文或大量段落复制进 `memory/`（只准用短引用 + 章节指向）。
- **不要**在 `memory/` 里写「产品战略最终结论」或推翻 MEMORY 的决策；若与 MEMORY 可能冲突，在手册里写 **「须与 docs/MEMORY.md 对齐后再执行」** 并列出疑点。
- **不要**凭空捏造仓库里不存在的路径或脚本名；不确定就写 **「待仓库确认」** 并列出需要人类核对的路径。

---

## 二、边界：memory/ 与 docs/MEMORY.md

| 内容类型 | 放哪里 |
|----------|--------|
| 产品方向、合规、国家站策略、架构原则、红线 | **只放在 `docs/MEMORY.md`**（由负责人改） |
| 具体命令、 checklist、流水线步骤、发布流程、排查步骤 | **`memory/` 下的 .md 文件** |

**优先级**：任何冲突以 **`docs/MEMORY.md`** 为准，`memory/` 仅作操作参考。

---

## 三、文件命名与存放位置

1. **目录**：一律在仓库根目录的 **`memory/`** 下（不要建 `memory/memory/`）。
2. **命名**：
   - 使用 **英文小写 + 连字符**（kebab-case），例如：`seo-afternoon-automation.md`、`deploy-vercel-checklist.md`。
   - 语义清晰：**主题 + 可选场景**，例如 `asset-seo-publish-queue-build.md`。
3. **扩展名**：**`.md`**（Markdown）。
4. **禁止**：不要用空格、中文文件名、版本号塞在文件名里（如 `v2-final.md`）；版本信息写在文件内的「修订记录」即可。
5. **模板**：新建文件时可对照 **`memory/_TEMPLATE-RUNBOOK.md`**，保证章节齐全。

## 四、文件格式（必须遵守）

### 4.1 整体

- 编码：**UTF-8**。
- 语言：**中文为主**（步骤、说明）；仓库内路径、命令、API 名保持 **原文**（英文）。
- 顶层标题：文件第一行使用 **一个** `# 标题`，与文件名主题一致、人类可读。

### 4.2 必须包含的章节（按顺序）

1. `# 标题` — 与内容一致。
2. `## 目的` — 1～3 句话：谁、在什么场景、解决什么问题。
3. `## 前置条件` — checklist（环境、分支、账号、`.env`、是否需 VPN 等）。
4. `## 步骤` — **编号列表**；每一步建议子结构（见第五节）。
5. `## 常见坑与回滚` — 至少一条「若失败则…」；没有则写「暂无，待补充」。
6. `## 相关路径` — 仓库内路径列表，用反引号包裹，例如 `` `scripts/build-asset-seo-publish-queue.ts` ``。
7. `## 规则依据` — **禁止粘贴 MEMORY 大段**。只写：「见 `docs/MEMORY.md` — 〈小节标题或关键词〉」。

### 4.3 建议包含的章节（按需）

- `## 输入与输出` — 脚本读写的文件、生成的 artifact。
- `## 验证 / 验收` — 如何确认成功（URL、JSON 字段、日志关键字）。
- `## 修订记录` — 日期 + 一行说明。

### 4.4 Markdown 习惯

- 命令放在 ** fenced code block ** 里，并标明 shell，例如：  
  ` ```powershell ` 或 ` ```bash `。
- 可选参数用方括号注明，例如：`[可选] --dry-run`。
- 内部链接：指向本仓库文件时用 **反引号路径**，例如 `` `memory/README.md` ``。

## 五、步骤怎么写（质量要求）

每一步 **至少**包含下面三类信息（可用小标题或加粗标出）：

1. **动作**：要做什么（一句人话）。
2. **命令或操作**：复制可运行的命令，或界面点击路径；无命令则写清楚「改哪个文件哪一段」。
3. **成功判据**：凭什么算这一步完成（命令退出码、文件出现、页面可打开等）。

**粒度**：宁可 **多步、每步短**，也不要一大段说不清。

**示例（一步的写法）**：

```markdown
2. **生成发布队列 artifact**
   - 在仓库根目录执行：`npx tsx scripts/build-asset-seo-publish-queue.ts`
   - 成功判据：生成 `generated/asset-seo-publish-queue.json`，且控制台无报错退出。
```

## 六、完成后你必须交付的两样东西

ChatGPT 在 **一次完整任务** 结束时应输出：

1. **新文件（或修改后文件）的完整 Markdown 正文**（或分文件逐个给出），便于用户粘贴进仓库。
2. **`memory/README.md` 里「索引」表格的新增行**（Markdown 表格的一行），格式严格为：

```markdown
| `your-file-name.md` | 一句话摘要（不超过 80 字） |
```

若用户要求你**同时改多个文件**，每个文件对应一行。

```

请产出：
1) memory/ 下新 Markdown 文件的完整内容（命名符合 kebab-case）；
2) memory/README.md 索引表需要追加的那一行；
3) 若与 docs/MEMORY.md 可能有冲突，单独列出「须人类确认」的条目。

```

（用户若未把仓库给 ChatGPT，需自行粘贴 `FOR-EXTERNAL-AI-CHATGPT.md` 的全文或关键节。）

## 八、Cursor / 其他 AI 在仓库里怎么用这些文件（按需自读）

**原则**：**需要哪里就自己读哪里**——由助手主动用工具打开文件，**不要**依赖用户预先粘贴 `memory/` 全文；用户只描述任务即可。

1. **不要**假设「已读完整个 `memory/`」，也**不要**在无关任务时通读所有手册。
2. **推荐流程**（助手自执行）：
   - Read **`memory/README.md`**，看索引表锁定与当前任务相关的文件名；
   - 若索引不够，用 **搜索**（如按关键词 grep / 代码库搜索）在 `memory/` 内找文件名或标题，再 **Read 命中文件**；
   - 通常每次任务 **1～3 个**相关 `.md` 足够；若手册指向某脚本/配置，再 Read 那些路径。
3. **规范与红线**：仍以 **`docs/MEMORY.md`** 为准（按 `AGENTS.md` 与 `.cursor/rules`，通常每会话首次实质性工作前读一次即可，不必每条用户消息重读）。
4. **与 ChatGPT（无仓库）的区别**：无仓库访问时，用户才需要把本文或某手册**粘贴**进对话；有仓库时以 **自读** 为主。
